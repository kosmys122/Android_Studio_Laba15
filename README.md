#  Student Planner

##  Описание приложения
Студенческий планер — это многоэкранное учебное приложение, разработанное для отслеживания дисциплин, текущих оценок и академической информации. Проект демонстрирует современные принципы построения UI и маршрутизации в Android с использованием декларативного подхода. Приложение обеспечивает плавные переходы между экранами, корректную работу системной кнопки «Назад» и безопасную передачу данных между маршрутами.

##  Реализованные экраны
- **HomeScreen** — главный экран со списком дисциплин в виде кликабельных карточек.
- **DetailsScreen** — экран детальной информации о выбранной дисциплине (преподаватель, кредиты, текущая оценка, описание курса).
- **ProfileScreen** — экран профиля студента с контактами, группой и статистикой успеваемости.
- **SettingsScreen** — экран настроек с переключателями (уведомления, тема) и навигационными пунктами меню.

##  Используемые технологии
- Язык: **Kotlin**
- UI-фреймворк: **Jetpack Compose** (Material 3)
- Навигация: **Navigation Compose** (`androidx.navigation:navigation-compose:2.7.6+`)
- Архитектура: **Unidirectional Data Flow** (передача данных через параметры и callbacks)

##  Схема навигации между экранами
- [HomeScreen]
- ├── (клик по карточке) ──> [DetailsScreen] (параметр: subjectId)
- ├── (кнопка "Профиль") ──> [ProfileScreen]
- └── (кнопка "Настройки") ─> [SettingsScreen]

# Ответы на контрольные вопросы
## 1. Что такое NavController и для чего он используется?
- Роль в навигации: NavController — это центральный управляющий компонент библиотеки Navigation Compose. Он отвечает за переходы между экранами (navigate()), управление стеком посещённых экранов (Back Stack), обработку системной кнопки «Назад» и передачу аргументов между маршрутами.
- Зачем rememberNavController(): Jetpack Compose постоянно вызывает рекомпозицию при изменении состояния. Если создать контроллер напрямую (NavController()), при каждой рекомпозиции будет создаваться новый экземпляр, что приведёт к потере истории переходов и сбросу навигации. rememberNavController() сохраняет экземпляр в памяти между рекомпозициями, гарантируя стабильную работу стека.
## 2. Как передать параметр в маршрут навигации?
Процесс:
- В маршруте объявляется плейсхолдер: "details/{subjectId}".
- В NavHost маршрут регистрируется с указанием типа: arguments = listOf(navArgument("subjectId") { type = NavType.StringType }).
- При вызове: navController.navigate("details/$id").
- Внутри composable-функции параметр извлекается: backStackEntry.arguments?.getString("subjectId").
- Обязательные vs опциональные: Обязательные параметры не имеют defaultValue и должны передаваться всегда, иначе приложение упадёт с ошибкой. Опциональные параметры задаются с defaultValue = "..." и могут быть опущены при вызове navigate(), маршрут всё равно сработает.
## 3. Зачем использовать sealed class для маршрутов?
- Преимущества: sealed class обеспечивает типобезопасность (type-safety). Компилятор знает все возможные экраны на этапе написания кода, что даёт автодополнение в IDE, позволяет использовать when без else и исключает использование «магических строк».
- Пример предотвращённой ошибки: При использовании обычных строк легко допустить опечатку: navController.navigate("detials"). Приложение крашнется с IllegalArgumentException. С Screen.Details.route IDE подсветит ошибку ещё до компиляции, а компилятор не позволит передать несуществующий маршрут.
## 4. Что такое Back Stack и как им управлять?
- Определение: Back Stack — это стек (LIFO: Last In, First Out) посещённых экранов. Он позволяет Android корректно обрабатывать кнопку «Назад».
- Схема для последовательности Home → Profile → Settings:
- ┌─────────────┐
- │  Settings   │ ← Текущий экран
- ├─────────────┤
- │   Profile   │
- ├─────────────┤
- │    Home     │ ← Стартовый экран
- └─────────────┘
- Вызов popBackStack() на Settings: Текущий экран (Settings) удаляется из стека, фокус переходит на Profile. При повторном вызове пользователь вернётся на Home. Если стек пуст, приложение закрывается.
## 5. Как работает startDestination в NavHost?
- Первый экран: startDestination определяет маршрут, который будет отображён сразу после инициализации NavHost при запуске приложения. В лабораторной работе это Screen.Home.route.
- Динамическое изменение: Формально параметр можно передать динамически (например, через аргумент функции), но на практике это делается редко. Изменение startDestination в runtime приводит к сбросу Back Stack и пересозданию навигационного графа, что может нарушить ожидаемое поведение кнопки «Назад». Обычно его задают статично при старте.
## 6. Что произойдёт, если навигировать на несуществующий маршрут?
- Реакция NavController: Выбросится IllegalArgumentException с сообщением navigation destination <route> is not a known destination, что приведёт к аварийному закрытию приложения (крашу).
- Как обработать:
- Использовать проверку перед вызовом navigate().
- Обернуть переход в try-catch и показать Snackbar/диалог с ошибкой.
- Добавить fallback-маршрут в NavHost (например, composable("error_route") { ErrorScreen() }) и перенаправлять на него неизвестные пути.
## 7. Зачем нужен параметр launchSingleTop в навигации?
- Пример проблемы без него: Если пользователь быстро нажмёт на одну и ту же карточку дисциплины 5 раз, в Back Stack добавятся 5 копий DetailsScreen. Чтобы вернуться на главный экран, ему придётся 5 раз нажать «Назад», что создаёт плохой UX.
- Влияние на стек: При вызове navController.navigate(route) { launchSingleTop = true } контроллер проверяет, находится ли целевой экран уже на вершине стека. Если да — навигация игнорируется (или обновляются аргументы без создания нового экземпляра). Это предотвращает дублирование экранов, делает Back Stack предсказуемым и снижает потребление памяти.
